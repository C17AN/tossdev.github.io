3. Tutorial

Toss API for Company 를 통해서 이용가능한 서비스는 은행 계좌주 성명 조회 및 지급 이체 가 있습니다.

이 문서를 통해서는 가상의 이벤트 회사인 My Event Company 를 통해서 서비스를 연동하는 과정을 알아보도록 하겠습니다.

3.1. 회사 소개

이벤트를 전문적으로 대행하는 My Event Company 에서는 다수의 이벤트를 대행하여 관리합니다. 응모자 중에서 당첨자에게는 당첨금액을 지급해주는데요, 이를 편리하게 관리하고자 백오피스에 연동하기 위해서 Toss API for Company 를 사용하고자 합니다.

3.2. 버전 조회

Toss API for Company 와 연동에 문제가 없는지 확인하기 위해 먼저 간단하게 확인을 합니다.

3.2.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/versions' -i -H 'Accept: application/json'
GET /fb/api/v1/company/versions HTTP/1.1
Accept: application/json
Host: toss.im

3.2.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 59

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 200
}

성공(200) 응답이 왔습니다. meta.version 값을 통해 현재 v1 버전을 쓰는걸 확인할 수 있습니다.

3.3. 모계좌 조회

My Event Company 모계좌 잔액 확인을 위해 요청을 합니다.

3.3.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/accounts' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0000'
POST /fb/api/v1/company/accounts HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0000

테스트 키 를 이용해서 모계좌 잔액을 확인해봅니다. (실제 발급된 테스트 키 를 ck_test_0000 대신 사용하면 됩니다.)

대부분의 요청에는 고객사별로 발급된 key 값이 필요합니다. 테스트용 키 와 실거래용 키 가 존재합니다. 테스트용 키 는 실거래가 이루어지지 않고, 결과를 반환합니다. 보통 개발시에 사용합니다.
파라미터에 대한 설명은 API 섹션에서 자세히 다루도록 하겠습니다.

3.3.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 99

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 200,
  "mode" : "TEST",
  "balance" : 10000
}

API 모드가 테스트(TEST) 이고 My Event Company 잔액이 10,000원 남아있는걸 확인했습니다.

3.4. 계좌주 성명 조회

이벤트에 당첨된 고객을 대상으로 당첨금액을 지급하기 위해, 성명 조회를 해봅니다.

3.4.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/bankaccounts' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0000&companyTxNo=company-tx-unique-0001&bankCode=0&bankAccountNo=0000000000000000'
POST /fb/api/v1/company/bankaccounts HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0000&companyTxNo=company-tx-unique-0001&bankCode=0&bankAccountNo=0000000000000000

은행 코드(bankCode) 와 은행 계좌 번호(bankAccountNo) 를 통해 계좌주 성명 조회가 가능합니다.

고객사 거래 번호(companyTxNo) 는 고객사에서 거래를 확인하기 위해 요청시 꼭 필요한 값입니다. 매 거래 요청에 대해 고유한 값을 주셔야합니다. API 섹션에서 부연 설명 드리겠습니다.

3.4.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 540

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 200,
  "mode" : "TEST",
  "account" : {
    "status" : "SUCCESS",
    "bankCode" : 0,
    "bankAccountNo" : "0000000000000000",
    "bankAccountName" : "Viva Lee",
    "message" : ""
  },
  "tx" : {
    "txNo" : "tx_test_255723da-8d65-449c-b416-3000da65c661",
    "companyTxNo" : "company-tx-unique-0001",
    "txType" : "ACCOUNT_LOOKUP",
    "txStatus" : "SUCCESS",
    "originTxNo" : "",
    "amount" : 0,
    "fee" : 40,
    "vat" : 4,
    "totalFee" : 44,
    "balance" : 9956
  }
}

다음의 몇가지 주요 내용을 확인할 수 있습니다.

  • 계좌주 성명 조회 결과(account.bankAccountName) 로 해당 은행계좌의 소유주가 Viva Lee 임을 확인했습니다.

  • 거래번호(tx.txNo) 를 통해 추후에도 거래 내역을 확인할 수 있습니다.

  • 수수료(tx.fee) 40원과 부가세(tx.vat) 4원이 발생하여, 거래 후 잔액(tx.balance) 이 9,956원임을 확인할 수 있습니다.

테스트 키 를 통해서 조회되는 계좌주 성명은 임의의 이름인 Viva Lee 로 고정적입니다. 라이브 키 를 통해서 조회시에는 파라미터로 전달한 은행 코드(bankCode) 와 은행 계좌 번호(bankAccountNo) 에 해당하는 계좌주의 성명이 조회됩니다.

3.5. 지급 이체

이벤트에 당첨된 고객의 계좌정보와 성명이 계좌주 성명 조회 결과와 일치하는걸 확인했으니, 당첨금을 지급하기로 합니다.

3.5.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/deposits' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0000&companyTxNo=company-tx-unique-0002&bankCode=0&bankAccountNo=0000000000000000&amount=600&withdrawAccountSummary=My+Event+Company&depositAccountSummary=Customer'
POST /fb/api/v1/company/deposits HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0000&companyTxNo=company-tx-unique-0002&bankCode=0&bankAccountNo=0000000000000000&amount=600&withdrawAccountSummary=My+Event+Company&depositAccountSummary=Customer

은행 코드(bankCode) , 은행 계좌 번호(bankAccountNo) 그리고 금액(amount) 를 통해 지급이체가 가능합니다.

통장 적요를 남기기 위해서는 다음의 두 파라미터를 사용합니다.

  • 집금이체 계좌 적요(withdrawAccountSummary) : 모계좌 적요. 실제로는 (주)비바리퍼블리카의 기업용 지급이체 모계좌에 남는 적요입니다.

  • 지급이체 계좌 적요(depositAccountSummary) : 받는분 계좌 적요

3.5.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 624

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 200,
  "mode" : "TEST",
  "deposit" : {
    "status" : "SUCCESS",
    "bankCode" : 0,
    "bankAccountNo" : "0000000000000000",
    "amount" : 600,
    "withdrawAccountSummary" : "My Event Company",
    "depositAccountSummary" : "Customer",
    "txReturnCallback" : ""
  },
  "tx" : {
    "txNo" : "tx_test_076b8fcd-ce5b-4483-9225-4dbead354dc0",
    "companyTxNo" : "company-tx-unique-0002",
    "txType" : "DEPOSIT",
    "txStatus" : "SUCCESS",
    "originTxNo" : "",
    "amount" : 600,
    "fee" : 200,
    "vat" : 20,
    "totalFee" : 220,
    "balance" : 9136
  }
}

응답에서 확인할 수 있는 주요 내용은 다음과 같습니다.

  • 거래 상태(tx.txStatus) : 성공(SUCCESS)

  • 기존에 계좌주 성명 조회를 마치고 난 잔액이 9,956원 이었는데, 지급이체 금액(tx.amount) 600원과 지급이체 수수료(tx.fee) 200원과 부가세(tx.vat) 20원이 발생하여, 거래 후 잔액(tx.balance) 이 9,136원임을 확인할 수 있습니다.

3.6. 입금 중 에러로 인한 환급 처리

3.6.1. 타행 이체 불능

'지급 이체' 요청이 '성공’으로 응답된 후에도 '입금 처리' 도중 에러가 발생하여 최종적으로 입금에 실패하고, 해당 금액이 환급되는 경우가 발생할 수 있어 이에 대한 처리가 반드시 필요합니다.

위 상황에 대한 처리를 위해, Toss는 '입금 중 에러로 인한 환급' 발생 시 Callback을 통해 가맹점에 변경 사항을 알립니다. (이체 생성 시 설정한 txReturnCallback값으로 전달)

이체 생성 시 txReturnCallback 값을 올바르게 설정하지 않으면, 입금에 최종적으로 실패하는 상황을 알 수 없게 됩니다. 원활한 처리를 위해 반드시 해당 값을 설정하고 에러 발생 시 적절히 대처하실 것을 권장합니다.

3.6.2. 파라미터

Parameter Description

txNo

거래번호

companyTxNo

고객사에서 발급한 거래번호

3.6.3. 응답

기대하지 않음 (미사용)

3.6.4. 발생 원인 및 대처

이체 요청이 '성공' 처리된 후 '타행 이체 불능’이 발생하는 과정은 아래와 같습니다.

  1. 이체 생성 시, Toss는 은행 공동망을 통해 이체를 요청하고 은행 공동망으로부터 '성공' 리턴을 받으면 Toss도 거래 '성공' 처리함

  2. 이체 금액은 은행 공동망을 통해 '받을 분’의 은행으로 이동하고, 받은 은행에서 입금 처리를 시작

  3. 입금 처리 과정에서 해당 계좌가 사용할 수 없는 상태 로 판명됨 (계좌 거래 정지, 동결 등)

  4. 받는 은행은 '타행 이체 불능' 에러를 발생시키고, 입금 금액을 은행 공동망으로 돌려보냄

  5. 입금 금액은 다시 Toss 충전금으로 환급되고, Toss는 가맹점에 환급 사실을 알림 (txReturnCallback)

    • '타행 이체 불능' 에러와 이에 따른 환급은 Toss 계좌이체 서비스 뿐만 아니라, 인터넷 뱅킹 등 통상적인 방법을 이용한 타행간 이체 시에도 발생할 수 있는 공통적인 에러입니다.

에러 발생 시 대처

  • 이체 불능 및 환급 처리 되는 경우 해당 계좌는 입금이 불가한 상태로 볼 수 있으며, 받을 분으로부터 대체할 다른 계좌 를 받아 다시 이체를 진행해야 합니다.

meta.version

String

API version

status

Number

API 호출 결과

- 200 : OK (성공)
- 400 : Bad Request (파라미터 오류)

- 409 : Conflict (중복 요청)
- 503 : Service Unavailable (은행 점검)
- 50x : Server Error (기타 서버 오류)

4.5. 거래 조회

API 는 기본적으로 요청과 그에 대한 응답으로 구성됩니다. 일반적인 경우라면 요청에 대해 바로 결과를 받을 수 있지만, 간혹 네트워크 상황에 문제가 있거나 하여 결과를 즉시 받지 못하는 경우에는 별도로 결과를 확인하는 과정이 필요합니다. 이를 위해 두 가지 거래 조회 API 가 존재합니다.

고객사 거래 번호 를 통해서 조회하는 방법은 거래에 대해 자동으로 생성되는 고유한 값인 거래 번호 를 받지 못했을 때 요청할 수 있는 방법입니다. (예를 들어, Toss 와의 통신 과정에서 문제가 발생한 경우)

거래 번호 를 통해 조회하는 방법은 요청에 대한 응답은 받았으나, 진행중 상태로 응답이 왔을 경우 결과 확인을 위해 요청할 수 있는 방법입니다. (예를 들어, Toss 와 은행과의 통신 과정에서 문제가 발생한 경우)

진행중(PROGRESS) 응답이 온 경우는 결과를 확인할 경우 다음과 같이 처리가 이루어집니다.

  • 진행중 (PROGRESS) : 거래 확인 중

  • 성공 (SUCCESS) : 거래 확인 결과 성공

  • 진행중 실패 (PROGRESS_FAIL) : 진행중 상태에서 거래 확인 결과 실패

진행중 환급 (PROGRESS_REFUND) : 진행중 상태에서 거래 확인 결과로 환급. 거래 요청시에는 고객사 모계좌의 한도(balance) 가 먼저 차감됩니다. 진행중 거래가 실패로 확인된 경우에는 한도를 복원해주기 위한 내역으로 '진행중 환급 상태’로 남게됩니다. 원거래 번호(originTxNo) 를 이용하여 연관된 거래를 찾을 수 있으며, 추후 제공될 거래내역 API 를 통한 조회시에 볼 수 있습니다.

5. Errors

Toss API for Company 를 통해서 서비스를 개발, 운영중에 만날 수 있는 대표적인 에러 케이스를 설명합니다.

5.1. ACL

인증 실패시에 발생합니다. 예를 들어 IP 접속 제한 을 1.1.1.1 에서만 가능하도록 신청했는데 2.2.2.2 에서 접근을 시도했을 때 발생합니다.

5.1.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/accounts' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0002'
POST /fb/api/v1/company/accounts HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0002

5.1.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 79

{
  "status" : 401,
  "ip" : "127.0.0.1",
  "message" : "Unauthorized Access"
}

5.2. 잘못된 API Key

잘못된 API Key 를 사용했을 때 발생합니다. 예를 들어 API Key 를 변경한 후, 구버전의 Key 를 사용했을 때 발생합니다.

5.2.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/accounts' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0001'
POST /fb/api/v1/company/accounts HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0001

5.2.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 92

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 404,
  "message" : "Invalid API Key"
}

5.3. 잘못된 거래 번호

잘못된 거래 번호(Tx No) 로 요청했을 때 발생합니다. 예를 들어 존재하지 않는 거래 번호를 통해 거래 확인 요청시에 발생합니다.

5.3.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/transactions' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0000&txNo=tx-unique-9999'
POST /fb/api/v1/company/transactions HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0000&txNo=tx-unique-9999

5.3.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 90

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 404,
  "message" : "Invalid Tx No"
}

5.4. 잘못된 파라미터

잘못된 파라미터로 요청했을 때 발생합니다. 예를 들어 형식에 맞지 않는 거래 번호로 요청시에 발생합니다.

5.4.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/transactions' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0000&txNo=so-far-far-very-very-very-very-long-long-long-long-long-long-long-transaction-no-parameter'
POST /fb/api/v1/company/transactions HTTP/1.1
Accept: application/json
Host: toss.im
Content-Type: application/x-www-form-urlencoded

key=ck_test_0000&txNo=so-far-far-very-very-very-very-long-long-long-long-long-long-long-transaction-no-parameter

5.4.2. 응답

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 298

{
  "meta" : {
    "version" : "v1"
  },
  "status" : 400,
  "message" : "Invalid Parameter",
  "parameter" : {
    "name" : "txNo",
    "value" : "so-far-far-very-very-very-very-long-long-long-long-long-long-long-transaction-no-parameter",
    "description" : "length should be less then 64"
  }
}

5.5. 기타 에러

예외 처리된 경우가 아닌 기대하지 않은 상항에서 오류가 발생한 경우입니다. 응답 파라미터에 있는 참조 ID(referenceId) 값을 통해 이슈 추적이 가능합니다.

5.5.1. 요청

$ curl 'https://toss.im/fb/api/v1/company/bankaccounts' -i -X POST -H 'Accept: application/json' -d 'key=ck_test_0000&companyTxNo=company-tx-unique-0001&bankCode=not-a-number&bankAccountNo=0000000000000000'
